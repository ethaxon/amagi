# 004-API

## 1. Document Purpose

This document defines amagi's API layering and major interface shapes. It is not the final OpenAPI document, but should serve as the design and implementation baseline for interfaces.

For database structure see `003-DATABASE.md`.
For sync semantics see `005-SYNC.md`.
For security and authentication see `008-SECURITY.md`.

---

## 2. API Layering

Three categories by purpose:

1. Dashboard API
2. Sync API
3. Auth / Security API

Recommended unified prefixes:

- `/api/v1/dashboard/...`
- `/api/v1/sync/...`
- `/api/auth/...`: OIDC, WebAuthn / authenticator, token-set, logout, and other auth endpoints bound to explicit protocol / security flows
- `/api/v1/auth/...`: Only for auth-adjacent business resources that may evolve significantly with business semantics; not used as the baseline path for OIDC / WebAuthn

---

## 3. Common Conventions

### 3.1 Data Format

- JSON request / response
- Times use ISO 8601 / RFC 3339
- IDs use string representation of UUID/ULID

### 3.2 Error Format

Recommended unified error response:

```json
{
  "error": {
    "code": "conflict_detected",
    "message": "A sync conflict was detected",
    "details": {}
  }
}
```

### 3.3 Paginated List Endpoints

Recommended support for:

- `limit`
- `cursor`

### 3.4 Versioning

Business resource endpoints are placed under `/api/v1/`, such as dashboard, bookmark, library, sync profile, sync preview/apply, etc. These endpoints' request/response shapes may change significantly with business model iteration and need a version prefix.

Authentication endpoints bound to explicit RFC / protocol flows do not use `/api/v1`; they are uniformly placed under `/api/auth/...` or protocol-required fixed paths, e.g., OIDC, token-set, WebAuthn / authenticator, logout. The semantics of these endpoints are primarily constrained by protocols and security flows, and should not change due to business API version evolution.

Do not expose business resource APIs like bookmark, library, sync, etc., under a path without a version prefix.

### 3.5 Runtime Health Checks

#### `GET /healthz`

Lightweight liveness endpoint.

Current response shape:

```json
{
  "status": "ok",
  "service": "amagi-api-server",
  "database": {
    "configured": false,
    "state": "not_configured",
    "auto_migrate": false,
    "detail": "database is not configured"
  }
}
```

Constraints:

- `healthz` does not fail because the database is not configured.
- `database.state` may only expose sanitized states such as `not_configured`, `connected`, `connection_failed`, `migration_failed`.
- Must not echo the database URL in `detail` or other fields.

#### `GET /readyz`

Readiness endpoint. Current behavior:

- Returns `200` when `database.state=ready`.
- Returns `503` when database is not configured, connection failed, auto-migrate failed, ping failed, or core schema is missing.

Current failure states at least include:

- `not_configured`
- `connection_failed`
- `migration_failed`
- `ping_failed`
- `schema_missing`

## 4. Dashboard API

The currently implemented dashboard business API uses `/api/v1/dashboard/...`. These endpoints must carry `Authorization: Bearer ...`; the OIDC source is preferentially sourced from the `X-Amagi-Oidc-Source` header, falling back to `default_oidc_source` when absent. Bearer token verification succeeding without an amagi account binding returns `401 unauthenticated`; no user is automatically created in business APIs. Route handlers are only responsible for request extraction / response mapping; bookmark read/write is delegated to `packages/amagi-bookmarks`.

### 4.1 Current User

#### `GET /api/v1/dashboard/me`

Returns the amagi user bound to the current bearer principal.

Example response:

```json
{
  "userId": "uuid"
}
```

### 4.2 Libraries

#### `GET /api/v1/dashboard/libraries`

Lists libraries visible to the user.

#### `POST /api/v1/dashboard/libraries`

Creates a normal library and returns the flat tree view of the library and root folder.

Example request:

```json
{
  "name": "Default",
  "kind": "normal"
}
```

Notes:

- Currently only supports `kind=normal`.
- `kind=vault` returns `vault_not_supported_in_iter6` and does not create a normal library.
- On successful creation, writes `libraries`, `library_heads`, root `bookmark_nodes` / `bookmark_meta`, `node_revisions` in the same transaction, and advances `currentRevisionClock` to `1`.

### 4.3 Tree

#### `GET /api/v1/dashboard/libraries/:libraryId/tree`

Returns the full tree as a flat adjacency list, not a nested tree.

Example response:

```json
{
  "library": {
    "id": "uuid",
    "kind": "normal",
    "name": "Default",
    "currentRevisionClock": 3,
    "createdAt": "2026-05-05T00:00:00Z",
    "updatedAt": "2026-05-05T00:00:00Z"
  },
  "nodes": [
    {
      "id": "uuid",
      "libraryId": "uuid",
      "parentId": null,
      "nodeType": "folder",
      "title": "Default",
      "sortKey": "root",
      "url": null,
      "urlNormalized": null,
      "isDeleted": false,
      "createdAt": "2026-05-05T00:00:00Z",
      "updatedAt": "2026-05-05T00:00:00Z"
    }
  ]
}
```

#### `POST /api/v1/dashboard/libraries/:libraryId/nodes`

Creates a node.

Example request:

```json
{
  "nodeType": "bookmark",
  "parentId": "uuid",
  "title": "Example",
  "url": "https://example.com"
}
```

Notes:

- `nodeType` supports `folder`, `bookmark`, `separator`.
- A normal create node must specify an undeleted folder parent within the same library.
- `bookmark` must have a non-empty URL; `folder` and `separator` cannot have a URL.
- `sortKey` is optional; when absent, the server generates an append-style sort key in the form `m{uuidv7}`.

#### `PATCH /api/v1/dashboard/nodes/:nodeId`

Updates node base fields.

#### `POST /api/v1/dashboard/nodes/:nodeId/move`

Moves a node.

Example request:

```json
{
  "parentId": "uuid",
  "sortKey": "m100"
}
```

Moving the root node, moving to self, or moving to a descendant returns a structured error.

#### `POST /api/v1/dashboard/nodes/:nodeId/delete`

Logically deletes a node.

#### `POST /api/v1/dashboard/nodes/:nodeId/restore`

Restores a node.

Currently, restore only restores the target node itself; subtree restoration strategy is left for subsequent sync/conflict iterations.

### 4.4 Revision Feed Baseline

#### `GET /api/v1/dashboard/libraries/:libraryId/revisions?afterClock=0&limit=100`

Returns a revision feed baseline available for dashboard/debug use. This endpoint is not the full sync preview/apply API.

Example response:

```json

  "toClock": 3,
  "nextCursor": null,
  "revisions": [

      "revId": "uuid",

      "payload": {},

    }
```

### 4.5 Metadata

#### `PATCH /api/v1/dashboard/nodes/:nodeId/meta`

Updates bookmark metadata.

Currently, Iter6 has not yet implemented this endpoint; this round only creates a baseline `bookmark_meta` row. The metadata patch API is left for subsequent dashboard/content enrichment iterations.

Example request:

## 5. Sync API

The sync API targets BrowserClient. See `005-SYNC.md` for semantic details.

The current Iter7 baseline implemented sync API, like the Dashboard API, uniformly uses the bearer principal baseline:

- Must carry `Authorization: Bearer ...`
- OIDC source preferentially from `X-Amagi-Oidc-Source`, falling back to `default_oidc_source`
- Bearer token verification succeeding without an amagi account binding returns `401 unauthenticated`
- Route handlers only do principal resolution, request extraction, and response mapping; business logic is delegated to `packages/amagi-sync`

### 5.1 Registration

#### `POST /api/v1/sync/clients/register`

Example request:

```json
{
  "device": {
    "deviceId": null,
    "device_name": "My PC",
    "device_type": "desktop",
    "platform": "windows"
  },
  "browser_client": {
    "browser_family": "chrome",
    "browser_profile_name": "Default",
    "extension_instance_id": "ext-123",
    "capabilities": {
      "can_read_bookmarks": true,
      "can_write_bookmarks": true
    }
  }
}
```

Current implementation highlights:

- `device.deviceId` is optional; creates a new device when absent, requires belonging to the current user when present
- `browser_clients` upsert by `(device_id, extension_instance_id)`
- Register advances `device.last_seen_at` and `browser_client.last_seen_at`

Response returns:

- `device`
- `browserClient`
- `defaultProfile`
- `cursorSummaries`

### 5.2 Session Start

#### `POST /api/v1/sync/session/start`

Input:

- `browser_client_id`
- Current profile intent
- Current local capability summary

Current implementation highlights:

- Guarantees at least one default manual profile exists per user
- Default profile is fixed as `mode=manual`, `defaultDirection=bidirectional`, `conflictPolicy=manual`
- Normal library defaults `projection=include`
- Vault library defaults `projection=excluded`

Output:

- `browserClient`
- `selectedProfile`
- `availableProfiles`
- `libraries`
- `cursors`
- `serverTime`

### 5.3 Feed Pull

#### `GET /api/v1/sync/feed`

Recommended query parameters:

- `browserClientId`
- `libraryId`
- `fromClock`
- `profileId`

Returns:

```json
{
  "browserClientId": "uuid",
  "libraryId": "uuid",
  "fromClock": 10,
  "toClock": 20,
  "currentClock": 20,
  "serverOps": [
    {
      "revId": "uuid",
      "logicalClock": 11,
      "opType": "node.update",
      "nodeId": "uuid",
      "payload": {}
    }
  ],
  "nextCursor": null
}
```

Notes:

- Feed reads directly from `node_revisions`, `fromClock` is an exclusive boundary
- `limit` defaults to 100, maximum 500
- Feed does not advance cursor; cursor can only be advanced via ack API
- Vault library returns `vault_sync_not_supported` and does not deliver vault content

### 5.4 Preview

#### `POST /api/v1/sync/preview`

Request:

```json
{
  "browser_client_id": "uuid",
  "profile_id": "uuid",
  "library_id": "uuid",
  "base_clock": 120,
  "local_snapshot_summary": {
    "root_hash": "hash"
  },
  "local_mutations": [
    {
      "client_mutation_id": "uuid",
      "op": "create",
      "parent_client_external_id": "b-100",
      "node_type": "bookmark",
      "title": "Example",
      "url": "https://example.com"
    }
  ]
}
```

Current implementation highlights:

- `clientMutationId` must be non-empty and unique within the same preview
- Mutation target can be resolved by `serverNodeId` or `clientExternalId`
- Create must provide `clientExternalId`; `node_client_mappings` is written after successful apply
- When `baseClock < currentClock` and local mutations exist, returns `stale_base_clock` conflict; no three-way merge in this round
- Preview persists to `sync_previews`; does not modify `bookmark_nodes`, `node_revisions`, or `sync_cursors`

Response:

```json
{
  "preview_id": "uuid",
  "summary": {
    "server_to_local": 3,
    "localToServerAccepted": 1,
    "conflicts": 1
  },
  "server_ops": [],
  "accepted_local_mutations": [],
  "conflicts": []
}
```

### 5.5 Apply

#### `POST /api/v1/sync/apply`

Example request:

```json
{
  "preview_id": "uuid",
  "confirm": true
}
```

Example response:

```json
{
  "applied": true,
  "newClock": 125,
  "serverOpsToApplyLocally": [],
  "createdMappings": [],
  "conflicts": []
}
```

Current implementation highlights:

- `confirm` must be `true`, otherwise returns `confirmation_required`
- Apply executes within a single DB transaction and reuses `packages/amagi-bookmarks`'s transaction-scoped mutation boundary
- Apply only consumes accepted mutations from `sync_previews`; does not trust re-submitted local deltas from the client
- For an already `applied` preview, repeated calls idempotently return the first apply result without duplicating nodes, revisions, or mappings
- If the library head clock does not match the preview's `toClock`, returns `preview_stale`

### 5.6 Cursor Ack

#### `POST /api/v1/sync/cursors/ack`

Example request:

```json
{
  "browserClientId": "uuid",
  "libraryId": "uuid",
  "appliedClock": 125,
  "lastAckRevId": "uuid"
}
```

Current implementation highlights:

- Ack only indicates that the browser has successfully applied to its local projection
- Requires `appliedClock <= library_heads.current_revision_clock`
- Ack is monotonic; if the request clock is less than the existing cursor, it does not regress, only returns the existing cursor
- Initial ack creates `sync_cursors` row; repeated ack of the same clock is idempotent

## 6. Auth / Security API

### 6.1 OIDC

#### `GET /api/auth/token-set/oidc/source/{source}/start`

Starts the amagi auth facade for a specified OIDC source. This path does not use `/api/v1` and is explicitly placed under the `token-set` namespace to avoid confusion with possible future cookie/session OIDC flows.

Iter5 current responsibilities:

- Delegates to `packages/amagi-securitydept`'s per-source backend-oidc runtime for SecurityDept login / authorize flow.
- Returns redirect status + headers produced by SecurityDept; does not manually write authorize URL, state, nonce, or PKCE in amagi route handlers.
- `source` must be consistent with the `oidc_sources` map key and should be threaded through pending state, callback, account binding, and subsequent audit.

#### `GET /api/auth/token-set/oidc/source/{source}/callback`

Fixed backend-oidc callback path. Currently delegates to SecurityDept backend-oidc `callback_fragment_return`, which returns fragment redirect semantics as an HTTP response. amagi does not manually write code exchange, metadata assembly, or token fragment concatenation in this handler.

#### `POST /api/auth/token-set/oidc/source/{source}/callback`

Fixed programmatic callback path. Currently delegates to SecurityDept backend-oidc `callback_body_return`, returning token-set and inline metadata in JSON body form.

#### `GET /auth/token-set/oidc/source/{source}/callback`

Frontend-oidc callback fixed path. Used by frontend app scenarios, no `/api` prefix. The current backend retains a typed shell response, clarifying that this path is consumed by the frontend app shell; the backend does not execute token exchange or session signing here.

#### `GET /api/auth/token-set/oidc/source/{source}/config`

Fixed frontend config projection path. Only returns frontend-safe fields such as source key, client id, scope, PKCE policy, fixed redirect/config path; must not return client secret, refresh token, raw token, or database URL.

#### `POST /api/auth/token-set/oidc/source/{source}/refresh`

Delegates to SecurityDept backend-oidc `refresh_body_return`, returning token delta and inline metadata in JSON body form.

#### `POST /api/auth/token-set/oidc/source/{source}/metadata/redeem`

Delegates to SecurityDept backend-oidc metadata redemption service. Unknown or expired redemption ids return a structured `404`.

#### `POST /api/auth/token-set/oidc/source/{source}/user-info`

Delegates to SecurityDept backend-oidc user-info service, providing `Authorization: Bearer ...` with access token and `id_token` in the request body. When the returned verified claims are sufficient to construct `ExternalOidcIdentity`, amagi continues to resolve / create account binding and returns in the response:

- `principal_resolution.status=resolved`
- or `principal_resolution.status=skipped_missing_claim`

Must not write raw access token, refresh token, or id token back to `claims_snapshot_json`.

Unknown source must return a structured `404`, with machine-facing code `auth.oidc.source_unknown`, not a panic or bare `500`.

Multiple OIDC sources must be explicitly expressed in the auth facade API using the provider key. Fixed path form `/api/auth/token-set/oidc/source/{source}/start` and the above token-set callback / projection paths are used; this `source` must be consistent with the config map key, pending state, `oidc_account_bindings.oidc_source`, and audit events.

Current facade handlers are only responsible for extraction / response mapping. Source resolution, config projection, verified-claims principal resolution, and account binding repository are located in `packages/amagi-auth`; `packages/amagi-securitydept` continues to handle SecurityDept typed config / runtime projection. A token-set authenticated principal only represents a base authentication identity and does not automatically grant vault library access.

#### `POST /api/v1/auth/vaults/:libraryId/unlock`


## 8. Authorization and Visibility

### 8.1 Dashboard API

Current baseline uses `Authorization: Bearer <access_token>` to resolve principal.

If a same-origin cookie session experience is needed in the future for the Dashboard Web UI, it can only be an optional session form at the frontend UX layer and must not change the bearer authentication contract of the business API.

### 8.2 Sync API

Requires a controlled authentication context bound to a registered browser client; this is not equivalent to a Dashboard Web cookie session.

### 8.3 Vault

Any API involving vault content must additionally verify:

- Base login identity
- Whether the corresponding library's unlock session is valid

## 9. Backward Compatibility Requirements

- Extension-facing API changes should be backward compatible where possible
- `preview/apply/ack` semantics must be changed cautiously once shipped
- New fields should generally be appended rather than breaking old client parsing
