# 004-API

## 1. 本文档目的

本文档定义 amagi 的 API 分层与主要接口形状。它不是最终 OpenAPI 文档，但应作为接口设计与实现基线。

数据库结构见 `003-DATABASE.md`。
同步语义见 `005-SYNC.md`。
安全与鉴权见 `008-SECURITY.md`。

---

## 2. API 分层

建议按用途分为三类：

1. Dashboard API
2. Sync API
3. Auth / Security API

统一前缀建议：

- `/api/v1/dashboard/...`
- `/api/v1/sync/...`
- `/api/auth/...`：OIDC、WebAuthn / authenticator、token-set、logout 等与明确协议 / 安全流程绑定的 auth endpoint
- `/api/v1/auth/...`：仅用于可能随业务语义大幅演进的 auth-adjacent 业务资源；当前不作为 OIDC / WebAuthn 基线路径

---

## 3. 通用约定

### 3.1 数据格式

- JSON request / response
- 时间统一使用 ISO 8601 / RFC 3339
- ID 使用字符串表示 UUID/ULID

### 3.2 错误格式

建议统一错误响应：

```json
{
  "error": {
    "code": "conflict_detected",
    "message": "A sync conflict was detected",
    "details": {}
  }
}
```

### 3.3 分页列表接口

建议支持：

- `limit`
- `cursor`

### 3.4 版本化

业务资源接口放在 `/api/v1/`，例如 dashboard、bookmark、library、sync profile、sync preview/apply 等。这类接口的 request / response shape 可能随着业务模型迭代发生较大变化，需要版本前缀。

与明确 RFC / 协议流程绑定的认证接口不使用 `/api/v1` 前缀，统一放在 `/api/auth/...` 或协议要求的固定 path 下，例如 OIDC、token-set、WebAuthn / authenticator、logout。这类接口的语义主要由协议和安全流程约束，不应因为业务 API 版本演进而改变 callback / challenge / assertion endpoint。

不要在无版本前缀下暴露 bookmark、library、sync 等业务资源 API。

### 3.5 运行时健康检查

#### `GET /healthz`

轻量 liveness endpoint。

当前返回形状：

```json
{
  "status": "ok",
  "service": "amagi-api-server",
  "database": {
    "configured": false,
    "state": "not_configured",
    "auto_migrate": false,
    "detail": "database is not configured"
  }
}
```

约束：

- `healthz` 不因为数据库未配置而失败。
- `database.state` 只允许暴露脱敏状态，例如 `not_configured`、`connected`、`connection_failed`、`migration_failed`。
- 不得在 `detail` 或其他字段中回显 database URL。

#### `GET /readyz`

readiness endpoint。当前行为：

- `database.state=ready` 时返回 `200`。
- 数据库未配置、连接失败、auto-migrate 失败、ping 失败或 core schema 缺失时返回 `503`。

当前失败状态至少包括：

- `not_configured`
- `connection_failed`
- `migration_failed`
- `ping_failed`
- `schema_missing`

## 4. Dashboard API

当前已实现的 dashboard 业务 API 使用 `/api/v1/dashboard/...`。这些接口必须携带 `Authorization: Bearer ...`；OIDC source 优先来自 `X-Amagi-Oidc-Source` header，缺失时使用 `default_oidc_source`。bearer token 验证成功但没有 amagi account binding 时返回 `401 unauthenticated`，不会在业务 API 中自动创建 user。route handler 只负责 request extraction / response mapping，bookmark 读写委托 `packages/amagi-bookmarks`。

当前 Iter11 开发期额外基线：

- API server 对 Dashboard dev origin 提供受限 CORS baseline
- 只允许 `http://localhost:4174` 与 `http://127.0.0.1:4174`
- 允许的 request headers 至少包括 `Authorization`、`X-Amagi-Oidc-Source`、`Content-Type`
- 这是开发态入口，不等同于生产部署策略

### 4.1 当前用户

#### `GET /api/v1/dashboard/me`

返回当前 bearer principal 绑定到的 amagi user。

示例响应：

```json
{
  "userId": "uuid"
}
```

### 4.2 Libraries

#### `GET /api/v1/dashboard/libraries`

列出用户可见的 libraries。

#### `POST /api/v1/dashboard/libraries`

创建 normal library，并返回 library 与 root folder 的 flat tree view。

示例请求：

```json
{
  "name": "Default",
  "kind": "normal"
}
```

说明：

- 当前仅支持 `kind=normal`。
- `kind=vault` 返回 `vault_not_supported_in_iter6`，不会创建普通 library。
- 创建成功时同一 transaction 内写入 `libraries`、`library_heads`、root `bookmark_nodes` / `bookmark_meta`、`node_revisions`，并将 `currentRevisionClock` 推进到 `1`。

#### `POST /api/v1/dashboard/libraries/:libraryId/nodes`

创建节点。

示例请求：

```json
{
  "nodeType": "bookmark",
  "parentId": "uuid",
  "title": "Example",
  "url": "https://example.com"
}
```

说明：

- `nodeType` 支持 `folder`、`bookmark`、`separator`。
- 普通 create node 必须指定同 library 内未删除的 folder parent。
- `bookmark` 必须有非空 URL；`folder` 与 `separator` 不能带 URL。
- `sortKey` 可选，缺失时服务端生成 `m{uuidv7}` 形式的追加 sort key。

#### `PATCH /api/v1/dashboard/nodes/:nodeId`

更新节点基础字段。

#### `POST /api/v1/dashboard/nodes/:nodeId/move`

移动节点。

示例请求：

```json
{
  "parentId": "uuid",
  "sortKey": "m100"
}
```

移动 root node、移动到自身或 descendant 下均返回结构化错误。

#### `POST /api/v1/dashboard/nodes/:nodeId/delete`

逻辑删除节点。

#### `POST /api/v1/dashboard/nodes/:nodeId/restore`

恢复节点。

当前 restore 只恢复目标 node 自身；子树恢复策略留到后续 sync/conflict 迭代。

### 4.7 Sync Profiles

#### `GET /api/v1/dashboard/sync-profiles`

返回当前 user 的 sync profile detail 列表。

当前 Iter11 行为：

- 若当前 user 还没有 profile，服务端会先确保存在一个 enabled manual default profile
- response 使用 camelCase JSON 字段
- `rules` 与 `targets` 始终内联返回，便于 Dashboard 第一屏直接编辑

示例响应：

```json
[
  {
    "id": "uuid",
    "name": "Default",
    "mode": "manual",
    "defaultDirection": "bidirectional",
    "conflictPolicy": "manual",
    "enabled": true,
    "rules": [
      {
        "id": "uuid",
        "ruleOrder": 10,
        "action": "include",
        "matcherType": "library_kind",
        "matcherValue": "normal",
        "options": {}
      },
      {
        "id": "uuid",
        "ruleOrder": 20,
        "action": "exclude",
        "matcherType": "library_kind",
        "matcherValue": "vault",
        "options": {}
      }
    ],
    "targets": []
  }
]
```

#### `POST /api/v1/dashboard/sync-profiles`

创建 profile。

当前 Iter11 约束：

- 只允许创建 `mode=manual`
- `defaultDirection` 只允许 `pull | push | bidirectional`
- `conflictPolicy` 只允许 `manual`
- 本轮不支持 profile hard delete，只支持更新和启停

示例请求：

```json
{
  "name": "Desktop Browsers",
  "mode": "manual",
  "defaultDirection": "bidirectional",
  "conflictPolicy": "manual"
}
```

#### `PATCH /api/v1/dashboard/sync-profiles/:profileId`

更新 profile 的可变字段。

当前只允许修改：

- `name`
- `enabled`
- `defaultDirection`
- `conflictPolicy`

若 PATCH 结果会让当前 user 没有任何 enabled manual profile，返回 `409 last_enabled_manual_profile`。

一旦 profile 被禁用，它不再能被 sync runtime 选中：

- `start_session()` 的 `availableProfiles` 只返回 enabled profiles
- `preferredProfileId` 若指向 disabled profile，返回 `400 profile_disabled`
- `feed()` / `preview()` 若传入 disabled `profileId`，同样返回 `400 profile_disabled`

#### `POST /api/v1/dashboard/sync-profiles/:profileId/targets`

新增 target selector。

当前 Iter11 约束：

- 至少一个 selector 字段必须非空
- `deviceId` 若存在，必须属于当前 user
- `browserClientId` 若存在，必须属于当前 user
- `platform` / `deviceType` / `browserFamily` trim 后为空则不会保存

示例请求：

```json
{
  "platform": "macos",
  "browserFamily": "firefox"
}
```

#### `DELETE /api/v1/dashboard/sync-profiles/:profileId/targets/:targetId`

删除 target selector。若 target 不属于当前 user 的 profile，返回 `404 target_not_found`。

#### `POST /api/v1/dashboard/sync-profiles/:profileId/rules`

新增规则。

当前 canonical matcher values：

- `library_kind`
- `folder_id`
- `folder_path`
- `tag`

示例请求：

```json
{
  "ruleOrder": 10,
  "action": "include",
  "matcherType": "folder_id",
  "matcherValue": "uuid",
  "options": {}
}
```

#### `PATCH /api/v1/dashboard/sync-profiles/:profileId/rules/:ruleId`

更新单条规则。

`matcherType` 的字段名是 camelCase，但其值仍然使用 canonical string，例如 `folder_path`，不是 `folderPath`。

#### `DELETE /api/v1/dashboard/sync-profiles/:profileId/rules/:ruleId`

删除规则。若 rule 不属于当前 user 的 profile，返回 `404 rule_not_found`。

## 5. Sync API

同步 API 面向 BrowserClient。详见语义说明 `005-SYNC.md`。

当前 Iter7 baseline 已实现的 sync API 与 Dashboard API 一样，统一使用 bearer principal baseline：

- 必须携带 `Authorization: Bearer ...`
- OIDC source 优先来自 `X-Amagi-Oidc-Source`，缺失时使用 `default_oidc_source`
- bearer token 验证成功但没有 amagi account binding 时返回 `401 unauthenticated`
- route handler 只做 principal 解析、request extraction、response mapping；业务逻辑委托 `packages/amagi-sync`

### 5.1 注册

#### `POST /api/v1/sync/clients/register`

示例请求：

```json
{
  "device": {
    "deviceId": null,
    "device_name": "My PC",
    "device_type": "desktop",
    "platform": "windows"
  },
  "browser_client": {
    "browser_family": "chrome",
    "browser_profile_name": "Default",
    "extension_instance_id": "ext-123",
    "capabilities": {
      "can_read_bookmarks": true,
      "can_write_bookmarks": true
    }
  }
}
```

当前实现要点：

- `device.deviceId` 可选；缺失时创建新 device，存在时要求属于当前 user
- `browser_clients` 按 `(device_id, extension_instance_id)` upsert
- register 会推进 `device.last_seen_at` 与 `browser_client.last_seen_at`

响应返回：

- `device`
- `browserClient`
- `defaultProfile`
- `cursorSummaries`

### 5.2 会话开始

#### `POST /api/v1/sync/session/start`

输入：

- `browser_client_id`
- 当前 profile 意向
- 当前本地能力摘要

当前实现要点：

- 保证每个 user 至少存在一个默认 manual profile
- 默认 profile 固定为 `mode=manual`、`defaultDirection=bidirectional`、`conflictPolicy=manual`
- normal library 默认 `projection=include`
- vault library 默认 `projection=excluded`

输出：

- `browserClient`
- `selectedProfile`
- `availableProfiles`
- `libraries`
- `cursors`
- `serverTime`

### 5.3 Feed 拉取

#### `GET /api/v1/sync/feed`

查询参数建议：

- `browserClientId`
- `libraryId`
- `fromClock`
- `profileId`

返回：

```json
{
  "browserClientId": "uuid",
  "libraryId": "uuid",
  "fromClock": 10,
  "toClock": 20,
  "currentClock": 20,
  "serverOps": [
    {
      "revId": "uuid",
      "logicalClock": 11,
      "opType": "node.update",
      "nodeId": "uuid",
      "payload": {}
    }
  ],
  "nextCursor": null
}
```

说明：

- feed 直接从 `node_revisions` 读取，`fromClock` 为 exclusive boundary
- `limit` 默认 100，最大 500
- feed 不推进 cursor；cursor 只能通过 ack API 推进
- vault library 返回 `vault_sync_not_supported`，不会下发 vault 内容

### 5.4 Preview

#### `POST /api/v1/sync/preview`

请求：

```json
{
  "browser_client_id": "uuid",
  "profile_id": "uuid",
  "library_id": "uuid",
  "base_clock": 120,
  "local_snapshot_summary": {
    "root_hash": "hash"
  },
  "local_mutations": [
    {
      "client_mutation_id": "uuid",
      "op": "create",
      "parent_client_external_id": "b-100",
      "node_type": "bookmark",
      "title": "Example",
      "url": "https://example.com"
    }
  ]
}
```

当前实现要点：

- `clientMutationId` 必须非空且在同一 preview 内唯一
- mutation target 可以通过 `serverNodeId` 或 `clientExternalId` 解析
- create 必须提供 `clientExternalId`，apply 成功后写 `node_client_mappings`
- `baseClock < currentClock` 且存在 local mutation 时返回 `stale_base_clock` conflict，本轮不做三方 merge
- preview 会持久化到 `sync_previews`；不会修改 `bookmark_nodes`、`node_revisions` 或 `sync_cursors`

响应：

```json
{
  "preview_id": "uuid",
  "summary": {
    "server_to_local": 3,
    "localToServerAccepted": 1,
    "conflicts": 1
  },
  "server_ops": [],
  "accepted_local_mutations": [],
  "conflicts": []
}
```

### 5.5 Apply

#### `POST /api/v1/sync/apply`

请求示例：

```json
{
  "preview_id": "uuid",
  "confirm": true
}
```

响应示例：

```json
{
  "applied": true,
  "newClock": 125,
  "serverOpsToApplyLocally": [],
  "createdMappings": [],
  "conflicts": []
}
```

当前实现要点：

- `confirm` 必须为 `true`，否则返回 `confirmation_required`
- apply 在单个 DB transaction 内执行，并复用 `packages/amagi-bookmarks` 的 transaction-scoped mutation boundary
- apply 只消费 `sync_previews` 中已接受的 mutation，不信任客户端重新提交的本地 delta
- 对已 `applied` 的 preview，重复调用会幂等返回首个 apply 结果，不重复写节点、revision 或 mapping
- 若 library head clock 与 preview 的 `toClock` 不一致，返回 `preview_stale`

### 5.6 Cursor Ack

#### `POST /api/v1/sync/cursors/ack`

请求示例：

```json
{
  "browserClientId": "uuid",
  "libraryId": "uuid",
  "appliedClock": 125,
  "lastAckRevId": "uuid"
}
```

当前实现要点：

- ack 只表示浏览器端已成功应用到本地 projection
- 要求 `appliedClock <= library_heads.current_revision_clock`
- ack 单调；若请求 clock 小于现有 cursor，不回退，只返回现有 cursor
- 初次 ack 创建 `sync_cursors` row；重复 ack 同一 clock 幂等

## 6. Auth / Security API

### 6.1 OIDC

#### `GET /api/auth/token-set/oidc/source/{source}/start`

启动指定 OIDC source 的 amagi auth facade。该路径不使用 `/api/v1`，并显式放在 `token-set` 命名空间下，避免与后续可能的 cookie/session OIDC flow 混淆。

Iter5 当前职责：

- 委托 `packages/amagi-securitydept` 的 per-source backend-oidc runtime 执行 SecurityDept login / authorize flow。
- 返回 SecurityDept 产生的 redirect status + headers，不在 amagi route handler 内手写 authorize URL、state、nonce 或 PKCE。
- `source` 仍必须与 `oidc_sources` map key 一致，并贯穿 pending state、callback、account binding 与后续 audit。

#### `GET /api/auth/token-set/oidc/source/{source}/callback`

backend-oidc callback 固定路径。当前委托 SecurityDept backend-oidc `callback_fragment_return`，返回 fragment redirect 语义的 HTTP response。amagi 不在该 handler 内手写 code exchange、metadata 组装或 token fragment 拼接。

#### `POST /api/auth/token-set/oidc/source/{source}/callback`

programmatic callback 固定路径。当前委托 SecurityDept backend-oidc `callback_body_return`，返回 JSON body 形式的 token-set 与 inline metadata。

#### `GET /auth/token-set/oidc/source/{source}/callback`

frontend-oidc callback 固定路径。前端 app 场景使用，无 `/api` 前缀。当前后端保留 typed shell response，明确该 path 由 frontend app shell 消费；后端不在这里执行 token exchange 或 session 签发。

#### `GET /api/auth/token-set/oidc/source/{source}/config`

frontend config projection 固定路径。只返回前端安全字段，例如 source key、client id、scope、PKCE policy、fixed redirect/config path；不得返回 client secret、refresh token、raw token 或 database URL。

#### `POST /api/auth/token-set/oidc/source/{source}/refresh`

委托 SecurityDept backend-oidc `refresh_body_return`，返回 JSON body 形式的 token delta 与 inline metadata。

#### `POST /api/auth/token-set/oidc/source/{source}/metadata/redeem`

委托 SecurityDept backend-oidc metadata redemption service。未知或过期的 redemption id 返回结构化 `404`。

#### `POST /api/auth/token-set/oidc/source/{source}/user-info`

委托 SecurityDept backend-oidc user-info service，`Authorization: Bearer ...` 中提供 access token，请求体提供 `id_token`。当返回的 verified claims 足够构造 `ExternalOidcIdentity` 时，amagi 会继续解析 / 创建 account binding，并在响应中返回：

Dashboard Web 与 extension 的 shared auth helper 会在本地 token-set bootstrap 成功后主动调用该接口，以完成 Dex `amagi/amagi` happy path 中的 account binding 建立或解析。

- `principal_resolution.status=resolved`
- 或 `principal_resolution.status=skipped_missing_claim`

不得把 raw access token、refresh token、id token 回写到 `claims_snapshot_json`。

unknown source 必须返回结构化 `404`，当前 machine-facing code 为 `auth.oidc.source_unknown`，而不是 panic 或空 `500`。

多 OIDC 来源必须在 auth facade API 中显式表达 provider key。固定使用 path 形式 `/api/auth/token-set/oidc/source/{source}/start` 和上述 token-set callback / projection path；该 `source` 必须与配置 map key、pending state、`oidc_account_bindings.oidc_source` 和 audit event 一致。

当前 facade handler 只负责 extraction / response mapping。source resolution、config projection、verified-claims principal resolution 与 account binding repository 位于 `packages/amagi-auth`；`packages/amagi-securitydept` 继续承接 SecurityDept typed config / runtime projection。token-set authenticated principal 只能表示基础认证身份，不能自动授予 vault library 访问权。

```json
  "unlock_session": {
    "id": "uuid",

  }

```

#### `POST /api/v1/auth/vaults/:libraryId/lock`

## 8. 授权与可见性

### 8.1 Dashboard API

当前基线使用 `Authorization: Bearer <access_token>` 解析 principal。

若未来 Dashboard Web UI 需要同域 cookie session 体验，也只能作为前端 UX 层的可选会话形态，不能改变业务 API 的 bearer 认证契约。

### 8.2 Sync API

需要已注册 browser client 绑定的受控认证上下文；这不等同于 Dashboard Web cookie session。

### 8.3 Vault

任何涉及 vault 内容的 API 都必须额外检查：

- base 登录身份
- 对应 library 的 unlock session 是否有效
